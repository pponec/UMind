The drawing is a two-sided mind map: root centred, up to 3 branches all go right, otherwise the first ‚åäN/2‚åã go right and the rest left, both in source order top to bottom (the left side is **mirrored, not rotated**). Labels are one line (measured with a canvas 2D context, truncated with `‚Ä¶` past `MAX_BOX_W`), columns widen so the widest box still leaves a readable connector, connectors are cubic B√©ziers, the palette is **always light** (print/share) regardless of the app theme, and `collapsed` is ignored ‚Äî everything is drawn expanded. A node's `note` becomes a UML-style bubble: a leaf's sits in the outer gutter level with its node, a parent's **hangs directly below its own node** flush with the node's inner edge (so it stays out of both the incoming connector's corridor and the child column, since `COL_MIN` > `NOTE_W`), and the root's goes in the free strip below the root. Layout runs in two passes, because the vertical one tests real rectangles and therefore needs x: **(1)** labels, widths, column offsets and x; **(2)** y. In pass 2 a node *plus its bubble, its connector corridors and its whole subtree* is a **rigid group**: leaves are handed out on a plain `SLOT` grid, a parent sits at the mean of its children, and each group then slides down only as far as an actual rectangle collision demands (`slideDown` against the side's occupancy list, the group's own rectangles excluded). That is the key to compactness ‚Äî a tall note costs height only in the column it actually occupies, so a branch can rise into the space *beside* a neighbour's long note instead of below it (on the welcome map a 15-line leaf note costs 6 px instead of 201 px). Connector corridors are reserved as `FAN_SLICES` pieces per link rather than one bounding box: the cubic is monotone in both axes, so each slice is exactly the curve's extent there and the staircase hugs the curve ‚Äî a whole-corridor box would cost ~70 px of needless height. Corridors test on strict overlap (`collides` drops the `NOTE_GAP` clearance when either rectangle is a corridor), otherwise every bubble flush with a column would count as sitting inside one. A bubble much wider than its node would sit in its *own* outgoing fan, which no collision test can catch (own rectangles are excluded), so `hangCrosses()` samples the link cubics against it ‚Äî using the narrowest column gap the drawing can still end up with, the worst case ‚Äî and that node falls back to a lane below its whole subtree, where no connector reaches. Bubble bodies are **real Markdown** ‚Äî the app's `renderMarkdown` output inside an SVG `<foreignObject>`, serialised with `XMLSerializer` (not `innerHTML`, whose `<br>`/`<img>` would break XML parsing) and measured in a hidden host so the reserved height is exact. **Nothing in a note may be clipped**, because a `<foreignObject>` silently cuts off whatever does not fit its rectangle: there is deliberately no maximum bubble height (an earlier `NOTE_MAX_H` of 300 px truncated long notes), code blocks get `white-space: pre-wrap` (a `pre` does not wrap by default, so a long line ran out of the bubble), long words/URLs/table cells get `overflow-wrap`, and the measured height carries `NOTE_SLACK` spare pixels in case the viewer's fonts wrap a line differently than ours did. The `<style>` block is wrapped in `/* <![CDATA[ */ ‚Ä¶ /* ]]> */`: the file is parsed as XML, where a raw `<` in the CSS starts a tag (a comment mentioning a `pre` tag broke every export once), while the markers sitting inside CSS comments keep the CSS valid when the SVG is inlined into an HTML page instead. Trade-off, accepted: `<foreignObject>` renders in browsers but **not** in Inkscape/librsvg or when rasterising to PNG. Layout runs on a private copy of the tree (`layoutTree`), so scratch coordinates never reach undo snapshots or localStorage. **Test maps** for the picture live in **`test/json/`** (five documents plus a README explaining what each one stresses: note sizes, five levels of nesting, mixed branch shapes, a note on every node, Markdown and escaping); load one with *Open‚Ä¶*. **The Maven build and the servlets do not exist yet** ‚Äî Phase 1 and 2 are not started.

## What UMind is

A minimalist, self-hosted mind-mapping app. Core insight: the browser already provides tree (DOM), layout (CSS), and editing (`contenteditable`) ‚Äî so **we write no rendering engine and no layout algorithm.** The app is an **outliner**: a nested list of editable nodes. Java does persistence only (accept JSON ‚Üí store; load JSON ‚Üí return); no map logic lives on the server.

## Locked decisions (do not re-litigate during build)

- **UI model:** outliner (nested `<ul>/<li>`), NOT free-positioned nodes on a canvas.
- **Node positioning:** computed by CSS/browser. No custom layout algorithm.
- **Text editing:** `contenteditable` per node; serialize to plain text.
- **Connector lines:** none in MVP ‚Äî hierarchy is shown by CSS indentation + vertical guide lines. SVG connectors are phase 2.
- **Frontend:** vanilla JS, no framework, no build step. One `.html` + `.js` + `.css`.
- **Java:** Jakarta Servlet (`jakarta.servlet`), plain `HttpServlet`.
- **Persistence:** whole document at once, debounced (500 ms) POST to one endpoint. No diffs. Storage = one JSON file per map, `{id}.json`, on disk. No database.
- **Undo/redo:** snapshot the full JSON state onto a stack. NOT a command pattern.
- **Auth:** none in MVP.

## Data model

Recursive tree via `children` (no flat node map in MVP). Node: `{ "id", "text", "note", "collapsed", "children": [] }`. `note` is an optional longer free-text description per node (added beyond the original spec), edited in a modal dialog (Alt+Enter on the focused node, or the detail panel's Edit button), trimmed on serialization, empty when unset. Nodes with a note show a üóí marker after their text ‚Äî a separate non-editable `.note-mark` span sharing a flex `.row` with the contenteditable `.node` (kept out of the editable so typing never displaces it; clicking it opens the description). The focused node's `note` is shown in a right-hand detail panel rendered as **Markdown** by an in-house renderer in its own file **`markdown.js`** (loaded before `app.js`; exposes the global `renderMarkdown(md)` ‚Üí HTML string, plus `renderMarkdownInto(parent, md)`). It is a JS port of Ujorm's `MarkdownToHtmlConverter` (`ujo-web`): headings 1‚Äì6, unordered/ordered lists, blockquotes, fenced and indented code blocks, horizontal rules, GFM tables, paragraphs with hard breaks (trailing `\`), and inline `code`/**bold**/_italic_/links/images. It builds the output as DOM nodes, so all text is escaped via `textContent` (no separate sanitization) and `javascript:`/`vbscript:`/`data:` link schemes are blocked. The Java `MarkdownToHtmlConverter` is intentionally *not* used on the server (the earlier "Java renderer" plan is dropped) ‚Äî this JS renderer is the single Markdown implementation. Document: `{ "version": 1, "rootId", "root": {‚Ä¶} }`. `id` is a short random string generated on the client (`"n_" + base36`). `collapsed` only controls display of children; data is retained.

## Build order (optimized for fastest running state)

- **Phase 0 ‚Äî clickable prototype, no Java.** Standalone HTML/JS/CSS: outliner, all keyboard shortcuts, undo/redo, in-memory JSON serialization (an "Export JSON" button into a textarea). **This is the largest part of the work** ‚Äî the interaction and edge cases are ironed out here before any backend exists.
- **Phase 1 ‚Äî persistence.** Add the three servlets; frontend does `load` on start and debounced `save` on change, replacing the in-memory export with API calls.
- **Phase 2 ‚Äî optional add-ons (separate iterations):** SVG connector overlay computed from `getBoundingClientRect()`, collapse/expand buttons, SVG/PNG export, `.mm` (FreeMind/Freeplane) import/export, node icons/colors.

## Keyboard model (core UX ‚Äî low-mouse)

`Enter` = new sibling below current + focus it; `Tab` = indent (become child of previous sibling); `Shift+Tab` = outdent (become sibling of parent); `‚Üë`/`‚Üì` = focus prev/next visible node; `Alt+‚Üë`/`Alt+‚Üì` = reorder the node among its siblings (no-op at the ends; added beyond the original spec); `Backspace` on empty node = delete, focus previous, reparent its children to the parent; `Ctrl+Z` / `Ctrl+Shift+Z` = undo/redo.

## Predecided edge cases (honor these ‚Äî they exist to save iterations)

- `Shift+Tab` on a node directly under root: no-op (don't crash).
- `Tab` on the first sibling (no previous sibling): no-op.
- Deleting the root: forbidden; root is always present.
- Empty node serializes as `text: ""` (allowed).
- Reading `contenteditable` text: use `element.innerText`, not `innerHTML`; trim, replace `¬Ý` with space, ignore injected `<br>`/`<div>`.
- Paste: intercept `paste`, insert plain text only (`clipboardData.getData('text/plain')`), prevent default HTML paste.
- IME/diacritics: don't act on `keydown` while `event.isComposing` is true.

## Explicitly out of scope (MVP)

Real-time collaboration, multi-user, cloud sync, mobile gestures, mouse drag-and-drop reorg (keyboard only for now), pixel positioning, custom layout algorithm, DB, auth.

## Code conventions (from the spec)

- **Build:** Maven project, Java 17, with a Maven Wrapper (`mvnw`) committed at the project root. Once scaffolded, typical commands: `./mvnw package`, `./mvnw test`, single test via `./mvnw test -Dtest=ClassName#method`.
- **Servlet implementation:** use the `Element` class from the `ujo-web` module of the Ujorm library ‚Äî see https://github.com/pponec/ujorm/blob/ujorm3/README_ELEMENT.md
- **Java:** use `var` for locals; English names and comments; JavaDoc comment above each method. Prefer `switch` over `if` where applicable.
- **Logging:** use the standard logging library; default level WARNING.
- **JS:** vanilla, no dependencies, English comments.
- **Single source of truth for the tree; render is a pure function of state (`renderTree(doc)`).**
